Embedding strategy

One vector per candidate. The profile text concatenates headline, skills, work experience (title + company + industry + description), education, languages, and location into a single document. The embedding model encodes all co-occurrences in one high-dimensional space.

def build_profile_text(row: dict) -> str:
    parts = [
        f"Name: {first} {last}",
        f"Professional Summary: {headline}",
        f"Years of Experience: {yoe}",
        f"Location: {city}, {country}",
        f"Skills: {', '.join(skills)}",
        f"Work Experience: {title} at {company} ({industry})",
        f"Education: {degree} in {field} from {institution}",
        f"Languages: {lang} ({proficiency})",
    ]
    return "\n".join(parts)

Why one vector, not per-field? Expert queries are holistic. "Regulatory affairs expert with pharma experience in Saudi Arabia" needs to match against role + industry + geography simultaneously. Per-field vectors require complex fusion logic without clear precision gains at this scale.

Precision Improvements & Future Architecture

Research-backed techniques that can improve the system's Precision@5 from ~0.70 to >0.85+, based on the latest published benchmarks (2024–2025).

Hybrid Search — BM25 + Dense Vectors

The current system uses dense-only vector search. Adding a sparse BM25 index alongside it creates a hybrid retrieval pipeline that consistently outperforms either alone.

📊
Benchmarked improvement: Hybrid search improves recall 15–30% over single methods. OpenAI + Qdrant benchmarks show recall increasing from 0.72 (BM25) → 0.91 (Hybrid) and precision from 0.68 → 0.87. Pinecone's Cascading Retrieval yields up to 48% better performance than dense-only. Source: Pinecone Research, 2024

How it works

BM25 matches exact tokens (critical for names, titles like "CPO", specific keywords). Dense vectors match semantics. Reciprocal Rank Fusion (RRF) merges both ranked lists without needing to normalise their different score scales:

def rrf_score(doc_id, bm25_rank, dense_rank, k=60):
    """k=60 is the standard constant from research papers."""
    score = 0
    if doc_id in bm25_results:
        score += 1 / (k + bm25_rank)
    if doc_id in dense_results:
        score += 1 / (k + dense_rank)
    return score

Implementation path

Pinecone now supports sparse+dense hybrid natively. Add sparse embeddings to the index, then query both simultaneously:

# pip install pinecone[grpc] rank-bm25
from pinecone_text.sparse import BM25Encoder

encoder = BM25Encoder().default()
sparse_vector = encoder.encode_documents([profile_text])[0]

# Upsert with both dense and sparse vectors
index.upsert(vectors=[{
    "id": candidate_id,
    "values": dense_embedding,    # existing
    "sparse_values": sparse_vector, # NEW
    "metadata": meta
}])

# Query with both
results = index.query(
    vector=query_dense,
    sparse_vector=query_sparse,
    alpha=0.5,   # 0=pure sparse, 1=pure dense
    top_k=top_k,
    include_metadata=True
)

When BM25 helps most for expert search

Cases where exact keyword matching is critical: queries containing exact company names ("Shell UK"), degree abbreviations ("MBA", "PhD"), specific skill names ("SPLADE", "GMP"), job title abbreviations ("CPO", "CFO"), and geographic abbreviations ("UAE", "KSA"). Dense vectors often mis-rank these because they depend on semantic overlap rather than exact match.

Neural Reranking — Cross-Encoders

The current re-ranker uses hand-crafted signal weights. A cross-encoder neural reranker evaluates each (query, candidate) pair jointly — capturing far more nuanced relevance signals than cosine distance.

📊
Pinecone reranking improves precision by reprocessing the top-K results with a cross-encoder model (pinecone-rerank-v0 or Cohere Rerank 3.5) that scores query-document relevance holistically. The cross-encoder architecture processes query and document together, capturing fine-grained relevance that bi-encoders miss. Source: Pinecone Blog, Dec 2024
from pinecone import Pinecone
pc = Pinecone(api_key=os.environ["PINECONE_API_KEY"])

# Stage 1: ANN retrieval (fast, broad)
initial_results = index.query(vector=query_vec, top_k=50)
documents = [m["metadata"]["profile_text"] for m in initial_results["matches"]]

# Stage 2: Neural rerank (precise, focused)
reranked = pc.inference.rerank(
    model="pinecone-rerank-v0",  # or "bge-reranker-v2-m3"
    query=query_text,
    documents=documents,
    top_n=10,
    return_documents=True
)

# Stage 3: Apply structured score adjustments on top
for item in reranked.data:
    neural_score = item.score
    adj, _ = seniority_score_adj(query, item.document["metadata"]["current_title"])
    final_score = neural_score + adj

Cascade: ANN (broad, fast) → Neural reranker (precise, slower) → Structured adjustments (interpretable). This "cascading retrieval" architecture is now Pinecone's recommended production pattern.

Neo4j Graph Database

Replacing or augmenting PostgreSQL with Neo4j enables relationship-aware expert search. The candidate database has natural graph structure: candidates connect to companies, skills, industries, institutions, and locations via typed edges.

🔗
Neo4j's graph model captures multi-hop relationships that relational joins cannot express efficiently: "find candidates who worked at a company whose client was Saudi Aramco, or who was trained by someone with 20+ years in the Middle East." These 2-3 hop queries are O(1) in a graph vs O(n·m) joins.

Graph schema for expert network

// Nodes
(:Candidate {id, name, headline, years_of_experience, country})
(:Company   {id, name, industry})
(:Skill     {id, name, category})
(:Institution {id, name})
(:Country   {id, name, code, region})
(:Language  {id, name})

// Relationships
(:Candidate)-[:WORKED_AT {title, start, end, is_current}]->(:Company)
(:Candidate)-[:HAS_SKILL {proficiency}]->(:Skill)
(:Candidate)-[:STUDIED_AT {degree, field, year}]->(:Institution)
(:Candidate)-[:LOCATED_IN]->(:Country)
(:Candidate)-[:SPEAKS {proficiency}]->(:Language)
(:Company)-[:OPERATES_IN]->(:Country)
(:Company)-[:IN_INDUSTRY]->(:Industry)

Precision gains from graph traversal

// Query: "Pharma regulatory experts who worked with Saudi Aramco-related companies"
MATCH (c:Candidate)-[:WORKED_AT]->(co:Company)-[:IN_INDUSTRY]->(:Industry {name:'Pharmaceutical'})
WHERE (co)-[:OPERATED_IN]->(:Country {name:'Saudi Arabia'})
  AND c.years_of_experience >= 10
MATCH (c)-[:HAS_SKILL]->(s:Skill)
WHERE s.name CONTAINS 'Regulatory'
RETURN c, collect(s.name) AS skills
ORDER BY c.years_of_experience DESC
LIMIT 20

Collaborative filtering via graph

// "Experts similar to candidate X based on shared career paths"
MATCH (a:Candidate {id: $candidate_id})-[:WORKED_AT]->(co:Company)
MATCH (b:Candidate)-[:WORKED_AT]->(co)
WHERE a <> b
WITH b, count(co) AS shared_companies
ORDER BY shared_companies DESC
LIMIT 10
RETURN b.name, b.headline, shared_companies

Graph Neural Networks (GNNs)

GNNs learn richer candidate representations by propagating information across the graph. A candidate's embedding is updated iteratively by aggregating features from their neighbouring nodes (companies, skills, colleagues) — capturing latent relationships that static embeddings miss.

📊
Documented improvements: Uber Eats achieved a 12% increase in AUC vs baseline using GNN-based recommendations. LightGCN and its variants are the de facto standard for graph-based recommendation. The key gain is capturing collaborative signals — candidates who have similar career trajectories cluster together in embedding space even if their profile texts differ. TDS Survey, Jan 2025

Message passing — how GNNs update embeddings

# Simplified GCN layer (PyTorch Geometric)
import torch
from torch_geometric.nn import GCNConv

class ExpertGNN(torch.nn.Module):
    def __init__(self, in_channels, hidden, out_channels):
        super().__init__()
        self.conv1 = GCNConv(in_channels, hidden)
        self.conv2 = GCNConv(hidden, out_channels)

    def forward(self, x, edge_index):
        # Layer 1: aggregate from immediate neighbours (companies, skills)
        x = self.conv1(x, edge_index).relu()
        # Layer 2: aggregate from 2-hop neighbours (colleagues, co-industries)
        x = self.conv2(x, edge_index)
        return x  # enriched candidate embeddings

# Then use these embeddings in Pinecone instead of text-only embeddings

Practical implementation path

1. Export candidate graph from PostgreSQL/Neo4j as edge lists. 2. Train a GNN to produce 256-dim candidate embeddings using career trajectory as supervision signal. 3. Replace text-only embeddings in Pinecone with GNN-enriched embeddings. 4. Fine-tune on historical engagement data (accepted shortlists) to align with business preferences.

GraphRAG Architecture

GraphRAG (Microsoft Research, 2024) combines knowledge graphs with LLM generation. Instead of treating candidate profiles as isolated text chunks, GraphRAG extracts entities and relationships, builds a hierarchical knowledge graph, and retrieves context by traversing graph structure — enabling multi-hop reasoning that flat vector search cannot do.

📊
Published results: GraphRAG achieves 72–83% comprehensiveness vs traditional RAG, with 3.4× accuracy improvement in enterprise scenarios. LazyGraphRAG (2025) reduces indexing cost to 0.1% of full GraphRAG while maintaining most gains. Microsoft Research, 2024–2025

Agentic RAG Chatbot Architecture

The current system is a classic RAG pipeline — one query → one retrieval → one answer. Agentic RAG transforms this into an autonomous multi-step reasoning system that can plan, retrieve from multiple sources, validate its own answers, and take follow-up actions.

💡
Agentic RAG vs Classic RAG: Classic RAG is a passive tool that responds to queries. Agentic RAG is a proactive system that decomposes complex queries into sub-queries, selects the best retrieval strategy per sub-query, executes them in parallel, validates the combined result, and iterates if confidence is low. Error rates drop by up to 78% in production agentic RAG systems (Cyfuture, 2025).

Future Roadmap — Prioritised by Impact

🔀 Hybrid BM25+Dense Search HIGH

Add sparse Pinecone index alongside existing dense. Use RRF fusion. Most immediate precision gain.

  • +15–30% recall improvement (benchmarked)
  • Critical for exact term queries (CPO, GMP, company names)
  • Pinecone native support — minimal code change
→ Pinecone hybrid search guide

🎯 Neural Reranker (Cross-Encoder) HIGH

Replace rule-based scoring with Pinecone rerank-v0 or Cohere Rerank 3.5. Captures holistic relevance.

  • Cross-encoder processes query+doc jointly
  • Works on top of any first-stage retrieval
  • Pinecone Inference API — no infrastructure
→ Pinecone reranking guide

🔗 Neo4j Graph Database MEDIUM

Add graph layer for relationship-aware queries. Multi-hop reasoning, collaborative filtering.

  • Career path similarity ("worked at similar companies")
  • Network effects ("colleagues of shortlisted experts")
  • Complex Cypher queries replace multi-JOIN SQL
→ Neo4j recommendation tutorial

🧠 Graph Neural Networks MEDIUM

Train GNN on career trajectory graph to produce richer candidate embeddings than text-only.

  • Captures latent collaborative signals
  • LightGCN proven for large-scale recommendation
  • Requires labelled engagement data for fine-tuning
→ GNN recommender tutorial (Decathlon)

🕸️ GraphRAG MEDIUM

Microsoft GraphRAG for multi-hop queries and community-level retrieval. Best for "find the Saudi pharma regulatory cluster".

  • 72–83% comprehensiveness vs flat RAG
  • 3.4× accuracy on enterprise scenarios
  • High indexing cost; use LazyGraphRAG variant
→ github.com/microsoft/graphrag

🤖 Agentic RAG Chatbot MEDIUM

Multi-tool agent: Router → Vector tool + SQL tool + Graph tool → Validator → Composer.

  • Decomposes complex queries automatically
  • Self-validates and iterates on low-confidence results
  • Long-term user preference memory via Zep/Mem0
→ IBM Agentic RAG guide

RAG-Anything — All-in-One Multimodal RAG

RAG-Anything (HKUDS / University of Hong Kong, 2025) extends traditional text-only RAG into a unified multimodal pipeline. Built on top of LightRAG, it processes PDFs, Office documents, images, tables, mathematical equations, and code — all through a single cohesive API. The repo has 15K+ GitHub stars and is actively maintained as of 2026.

🌟
Why it matters for expert search: Candidate profiles in the real world come as PDFs, CVs with embedded charts, LinkedIn exports, scanned documents, and research publications with equations. RAG-Anything can process all of these and build a unified multimodal knowledge graph — so a query like "Find candidates who published work on GNN-based recommendation systems" can retrieve evidence from both the text and the figures in those papers.

Core architecture — 5-stage pipeline

StageWhat happensKey component
1. Document ParsingMinerU / Docling / PaddleOCR extracts text blocks, images, tables, equations with layout preservationMinerU integration
2. Content RoutingAutonomous categorisation routes each element to its specialised processor concurrentlyConcurrent multi-pipeline
3. Multimodal AnalysisVision LLM captions images; table interpreter extracts trends; LaTeX parser handles equationsImageModalProcessor, TableModalProcessor, EquationProcessor
4. Knowledge Graph IndexEntities + cross-modal relationships extracted and scored; hierarchical "belongs_to" chains preservedLightRAG KG layer
5. Modality-Aware RetrievalVector-Graph fusion with modality-aware ranking and relational coherence maintenanceHybrid retrieval engine

Supported content types

Documents: PDF, DOC/DOCX, PPT/PPTX, XLS/XLSX, TXT, MD  ·  Images: JPG, PNG, BMP, TIFF, GIF, WebP  ·  Modalities: text, images, tables, equations (LaTeX), custom types via plugin API

The result is a richer knowledge graph where entities like "Pharmaceutical Regulatory Affairs" are linked to both textual mentions and visual evidence (certificate scans, org charts).

MMGraphRAG — Multimodal Knowledge Graphs with Vision + Language

MMGraphRAG (Wan & Yu, arXiv:2507.20804, 2026 — Harbin Institute of Technology / NTU Singapore) extends GraphRAG into the multimodal domain. Where classic GraphRAG builds a knowledge graph from text alone, MMGraphRAG constructs an interpretable multimodal knowledge graph that fuses visual entities (extracted from images via a dedicated Image2Graph module) with textual entities through a cross-modal fusion module.

📄
Key innovation: MMGraphRAG introduces two novel components — Image2Graph (converts image content into structured graph entities and relationships) and SpecLink (a spectral clustering-based entity alignment method that matches visual entities to their textual counterparts with fine-grained precision). Together they enable true cross-modal reasoning: a query about a "pharmaceutical manufacturing process" can retrieve both textual descriptions and visual flowcharts that depict it.

Architecture — three-module pipeline

ModuleRoleTechnical detail
Image2GraphConverts image content into structured KG entities and typed edgesVLM extracts visual entities; spatial relationships mapped to graph edges; entity metadata preserved
SpecLinkAligns visual entities to textual entities in the existing KGSpectral clustering generates candidate matches; CMEL (Cross-Modal Entity Linking) task for fine-grained alignment; iterative graph construction
Cross-Modal FusionMerges aligned entity pairs into unified representations for retrievalEntity fusion for unified representation; weighted relevance scoring; relational coherence across modalities

How it differs from RAG-Anything

DimensionRAG-AnythingMMGraphRAG
Primary goalUniversal document processing — handle any modalityInterpretable multimodal KG — explain cross-modal reasoning
Graph constructionLightRAG-based KG with multimodal entity nodesSeparate visual KG + textual KG → fused via SpecLink alignment
Entity alignmentImplicit (VLM captions bridge modalities)Explicit spectral clustering-based CMEL task
InterpretabilityGood (entity extraction visible)High (alignment scores + spectral evidence for every cross-modal link)
Best forDiverse document corpora, operational pipelinesResearch requiring explainable cross-modal reasoning

Relevance to expert network search

For expert search, MMGraphRAG's cross-modal fusion approach means that:

  • A candidate's textual claim ("expertise in GMP pharmaceutical manufacturing") can be aligned to visual evidence — certification images, org chart positions, publication figures — increasing retrieval confidence
  • The SpecLink alignment produces interpretable confidence scores for each text-image link, enabling explainability ("this candidate ranked highly because their textual bio aligns with their certification image with confidence 0.91")
  • Queries involving both text and image intent ("find candidates who appear in pharma plant photographs AND have regulatory experience") become tractable via unified KG traversal

# Current system: text-only profile → one dense vector → Pinecone ANN
profile_text = build_profile_text(candidate)

# MMGraphRAG upgrade path (conceptual):
# 1. Extract visual entities from CV images (Image2Graph)

# 2. Build textual KG from profile text (existing LightRAG approach)
text_kg    = lightrag.build_kg(profile_text)

# 3. Align visual ↔ textual entities (SpecLink)
fused_kg   = speclink.align(visual_kg, text_kg, method="spectral_clustering")

# 4. Unified KG retrieval with cross-modal evidence scoring
results    = mmgraphrag.query(
    query="Regulatory expert with pharma certifications Saudi Arabia",
    kg=fused_kg
)
# Each result includes: relevance_score + modal_evidence + alignment_confidence

Combined multimodal future architecture

Combining RAG-Anything (operational processing) with MMGraphRAG (cross-modal reasoning) and Neo4j (relational graph) produces the most complete expert search system:

Layer 1 — Ingestion: RAG-Anything parses all CV formats → text + images + tables + equations
Layer 2 — Representation: MMGraphRAG Image2Graph + SpecLink → fused multimodal KG per candidate
Layer 3 — Storage: Neo4j stores relational graph · Pinecone stores dense embeddings · BM25 stores sparse index
Layer 4 — Retrieval: Hybrid BM25+Dense → Neural reranker → Graph traversal → Cross-modal fusion score
Layer 5 — Generation: Agentic RAG composes answer with textual + visual + relational evidence
